# Notikeep — Продуктовый план

> База для решений — [RESEARCH.md](./RESEARCH.md). Архитектура — [ARCHITECTURE.md](./ARCHITECTURE.md).

---

## 1. Что делает приложение (в одну фразу)
**Notikeep сохраняет каждое уведомление в локальный архив и гасит мусор из шторки по
простым правилам пользователя — так, что ничего не теряется.**

Функции:
- Перехват всех уведомлений → сохранение в локальную БД (Room).
- Гашение мусорных пушей в шторке по правилам пользователя (но в архиве они остаются).
- Архив — лента, сгруппированная по приложениям.
- Поиск по тексту за весь срок хранения.
- Правила на приложение: `Шторка + Архив` / `Только архив` / `Игнорировать`.

---

## 2. УТП (три честных пункта)
1. **Первый в нише на русском в RuStore.** Прямых аналогов нет (см. ресёрч).
2. **Без рекламы и без облака.** Всё локально. Анти-Notisave.
3. **Проще BuzzKill.** Поставил — работает — ничего не пропало. Три состояния вместо конструктора правил.

## 3. Границы MVP

**Входит в MVP:**
- Онбординг (ценность + честное «сохраняем с этого момента» + доступ + battery).
- Захват уведомлений через `NotificationListenerService`.
- Архив-лента с группировкой по приложению + bottom-sheet детали.
- Поиск по тексту.
- Правила на приложение (3 состояния) + фактическое гашение.
- Настройки: тема, срок хранения, статус сервиса, экспорт.
- **Аналитика поведения** (обезличенная, локально-агрегируемая) — событийная модель с первого дня.
- Индикатор здоровья фонового сервиса.

**НЕ входит в MVP (осознанно отложено):**
- Монетизация/биллинг — только интерфейс-заглушка `EntitlementProvider`, реализации нет.
- Восстановление удалённых сообщений / автосейв медиа и статусов (фичи Notisave — юридически/приватностно спорные).
- Конструктор сложных правил по ключевым словам (анти-BuzzKill).
- Облачный бэкап (принципиально — только локальный экспорт).
- Виджеты, iOS.

## 4. Карта экранов и путь пользователя

```
Онбординг ──> Архив (главный) ──┬──> Поиск
                                ├──> Приложения/Правила
                                └──> Настройки
   (детали уведомления = bottom-sheet поверх Архива)
```

| Экран | Задача | Ключевые состояния |
|---|---|---|
| **Онбординг** | Объяснить, выдать доступ, отключить battery opt | пусто/доступ выдан ✓ |
| **Архив** | Лента всех уведомлений по приложениям | загрузка / пусто (дружелюбный) / данные |
| **Поиск** | Найти по тексту за всё время | пусто-запрос / нет результатов / результаты |
| **Правила** | 3 состояния на приложение | список приложений с сегмент-контролом |
| **Настройки** | Тема, срок хранения, статус сервиса, экспорт | — |

Empty-state Архива в первый день — **дружелюбный текст**, повторяющий обещание
онбординга («Сохраняем с этого момента»), чтобы убить жалобу «удалило всё» (см. антипаттерн №1 в ресёрче).

## 5. Аналитика поведения (обязательно с первого дня)

Цель — понимать путь пользователя: где отваливается онбординг, доходит ли до
выдачи доступа, каким экраном пользуется, гасит ли приложения.

Принципы (чтобы не повторить провал Notisave по приватности):
- **Обезличенно.** Никакого содержимого уведомлений в аналитике — только события UX.
- **Прозрачно.** Экран настроек: тумблер «Помогать улучшать приложение» + описание.
- **Абстрагировано.** Интерфейс `Analytics` в domain-слое; реализация подменяема
  (сейчас — локальный лог/агрегатор, позже — RuStore-совместимый провайдер).

Минимальный событийный словарь MVP:
- `onboarding_step_viewed(step)`, `notification_access_granted`, `battery_opt_disabled`
- `archive_opened`, `search_used`, `notification_detail_opened`
- `rule_changed(state)` — без имени приложения-персоналии, только состояние
- `service_health(status)`

## 6. Приоритеты реализации (порядок работ)
1. Фундамент: Kotlin + Compose + Room + Hilt + слои (см. ARCHITECTURE.md).
2. `NotificationListenerService` → запись в Room (сердце продукта).
3. Архив-лента + детали.
4. Правила на приложение + гашение.
5. Поиск.
6. Онбординг + индикатор здоровья сервиса.
7. Аналитика (интерфейс с шага 1, события навешиваем по ходу).
8. Настройки + экспорт.
