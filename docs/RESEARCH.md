# Notikeep — Ресёрч конкурентов и UX

> Цель документа: понять, из каких экранов состоит хорошее приложение-архив
> уведомлений, какую задачу закрывает каждый экран, как пользователь с ним
> взаимодействует, и на каких ошибках конкурентов мы **не** должны повторяться.
>
> Источники — веб-ресёрч по Google Play / App Store / обзорам (июль 2026).
> Прямого доступа к RuStore-листингам из ресёрч-сессии не было, поэтому выводы
> о нише RuStore строятся на общей картине рынка + отсутствии прямых аналогов
> на русском.

## 1. Карта конкурентов

| Приложение | Ниша / фокус | Сильное | Слабое (по отзывам) |
|---|---|---|---|
| **Notification History Log** (ikvaesolutions) | Эталон «архива»: 4.0★, ~39K отзывов | Advanced History экран, мощный поиск, тёмная/светлая тема, виджет, **шифрованный локальный бэкап**, лёгкий по батарее | Платная версия дорогая; проблемы с удалением отдельных записей |
| **Notisave** (tenqube) | «Читалка сообщений» + сохранение статусов WhatsApp | Чтение без «прочитано», восстановление удалённых сообщений, автосейв картинок/статусов | **Агрессивная реклама** (подписка $18.99/нед ради снятия рекламы), теряет групповые сообщения, не тянет медиа/голосовые, приватность (широкие storage-права, нет прозрачности данных), в обновлении сломали клик по уведомлению |
| **BuzzKill** | Правила/автоматизация уведомлений | Гибкие правила: фильтр, мьют по ключевым словам, авто-dismiss, действия по контенту, снуз | **Высокий порог входа** (правила неинтуитивны), правила не срабатывают из-за battery optimization, история ограничена, UI менее полирован |
| **FilterBox** | AI-фильтр спам-уведомлений | Offline-AI фильтр, timeline истории (view/copy/restore), статистика по дням, отпечаток-локк, темы (Android/iOS-стиль) | Часть функций за разовой покупкой (30-дн триал) |
| **Norg** (Lufesu) | Фильтр + история | Сортирует важное/неважное, история для просмотра позже | **Конфликтует с другими notif-менеджерами**; требует явного исключения из battery optimization, иначе сервис умирает |
| **NotificationLog** (OSS) | Приватный лёгкий лог | Бесплатный, open-source, экспорт, приватность | Минимум фич, простой UI |

### Вывод по нише
- **Прямого русскоязычного конкурента с фокусом «архив + гашение мусора» в RuStore нет** — это подтверждает УТП «первый в нише».
- Рынок делится на два лагеря: **«архиваторы»** (History Log, NotificationLog — сохраняют всё, но не гасят) и **«фильтры/правила»** (BuzzKill, Norg — гасят, но история слабая и настройка сложная).
- **Notikeep занимает пустую середину**: сохраняет ВСЁ + гасит мусор + простая настройка. Это и есть продуктовое отличие, а не просто «ещё один лог».

## 2. UX-путь пользователя (синтез лучших практик)

Пользователь такого приложения проходит ровно 3 фазы. Экраны нужно проектировать под них.

### Фаза A — Первый запуск (онбординг + выдача доступа)
Самая хрупкая фаза. Здесь конкуренты теряют пользователей.
- Приложение бесполезно без **Notification Access** (`NotificationListenerService`).
- Пользователь не понимает, зачем такое «страшное» разрешение → бросает.
- **Батарейная оптимизация** убивает listener → «перестало работать» (жалобы на Norg/BuzzKill).

### Фаза B — Ежедневное использование (архив + поиск)
- «Пропустил уведомление / собеседник удалил сообщение — хочу посмотреть».
- Основной сценарий: открыл → увидел ленту → нашёл нужное (поиск/группировка по приложению).
- Должно быть **быстро и без когнитивной нагрузки** (антипример — сложность BuzzKill).

### Фаза C — Настройка тишины (правила гашения)
- «Это приложение спамит — не показывай его в шторке, но сохраняй в архив».
- Ключевое отличие Notikeep: **гашение ≠ потеря**. Погашенное всё равно в архиве.
- Настройка должна быть **на уровне приложения одним тапом** (3 состояния), а не конструктором правил как в BuzzKill.

## 3. Рекомендуемая карта экранов

Минимально достаточный набор — **4 экрана + онбординг**. Больше не нужно: лишние
экраны = сложность = антипаттерн BuzzKill.

| # | Экран | Задача (что закрывает) | Взаимодействие |
|---|---|---|---|
| 0 | **Онбординг** (2–3 шага) | Объяснить ценность, честно предупредить «сохраняем только с этого момента», выдать Notification Access + отключить battery optimization | Свайп по шагам, кнопки перехода в системные настройки, индикатор «доступ выдан ✓» |
| 1 | **Архив (лента)** — главный | Показать все уведомления, сгруппированные по приложению; быстрый доступ к недавнему | Скролл, тап по группе → раскрытие, тап по уведомлению → детали/оригинальное приложение, свайп → удалить |
| 2 | **Поиск** | Найти любое уведомление по тексту за всё время | Строка поиска, live-результаты, фильтр по приложению/дате |
| 3 | **Приложения / Правила** | Для каждого приложения задать поведение: `Шторка+Архив` / `Только архив` / `Игнорировать` | Список приложений с иконками, сегмент-контрол из 3 состояний на строку |
| 4 | **Настройки** | Тема, срок хранения, экспорт/бэкап, статус доступа, «О приложении» | Обычный список настроек |

> Детальный экран уведомления делаем не отдельным окном, а bottom-sheet поверх
> ленты — меньше навигации, современнее, ближе к материал-паттернам.

## 4. Антипаттерны конкурентов → наши правила (что НЕ делать)

Это самый ценный раздел. Каждый пункт — прямая причина негативных отзывов у конкурентов.

1. **«Удалило все мои уведомления при первом запуске».**
   Реальная причина: сервис ловит уведомления **только с момента выдачи доступа**, а
   старые из системы он взять не может. Пустой лог пользователь воспринимает как
   «удалили».
   → **Наше правило:** на онбординге честно и заметно писать: *«Notikeep начинает
   сохранять уведомления с этого момента. Показать те, что пришли раньше установки,
   технически невозможно»*. Пустой архив в первый день — с дружелюбным empty-state,
   а не с молчаливым пустым списком.

2. **Агрессивная реклама и абсурдные подписки** (Notisave — $18.99/нед за снятие рекламы).
   → **Наше правило:** ноль рекламы. Это прямое УТП. (Монетизация позже и разовая.)

3. **Слив данных / широкие права / нет прозрачности** (Notisave — storage-права, непрозрачная политика).
   → **Наше правило:** всё локально, минимум разрешений, честная privacy-policy,
   никакого облака. Аналитику поведения (которую мы закладываем) держим
   **обезличенной и локально-агрегируемой**, с прозрачным описанием — иначе повторим
   ошибку Notisave.

4. **Listener умирает от battery optimization** (Norg, BuzzKill — «перестало работать»).
   → **Наше правило:** в онбординге явный шаг «исключить из оптимизации батареи» +
   на главном экране индикатор здоровья сервиса («Слежение активно ✓ / Требует
   внимания ⚠»), чтобы пользователь сразу видел проблему, а не узнавал через неделю.

5. **Сложность настройки правил** (BuzzKill — «правила не срабатывают, неинтуитивно»).
   → **Наше правило:** никакого конструктора правил в MVP. Только 3 понятных
   состояния на приложение. Сложные правила — потом, опционально, не ломая простоту.

6. **Сломали привычный сценарий обновлением** (Notisave — убрали клик по уведомлению).
   → **Наше правило:** тап по уведомлению = открыть оригинальное приложение/действие.
   Это базовый ожидаемый жест, не трогаем его.

7. **Конфликт с другими notif-менеджерами** (Norg).
   → **Наше правило:** не пытаться быть единственным listener агрессивно; корректно
   сосуществовать, гасить только то, что пользователь явно пометил.

## 5. Итоговые продуктовые выводы

- **Формула Notikeep = архив (History Log) + простое гашение (лучшее от BuzzKill без сложности) − реклама/облако (анти-Notisave).**
- Экранов ровно 4 + онбординг. Простота — это фича, а не недоработка.
- Самый большой риск-провал — онбординг и «пустой первый запуск». Вкладываемся в него отдельно.
- Здоровье фонового сервиса нужно делать **видимым** пользователю.
- Аналитику поведения закладываем с первого дня, но обезличенно и прозрачно.
